01设计理念

🏠

桌面应用体验，Web 技术栈

单页应用 (SPA)，python app.py 一键启动后自动打开浏览器。无构建步骤、无 npm、无打包 — 开箱即用。

🧩

模块化拆分，按功能解耦

后端 11 个 Blueprint + 前端 13 个 JS/13 个 CSS 文件，每个模块只负责一件事。新增功能只需加文件、注册路由、引入脚本。

🧠

双轨分析：规则引擎 + AI

规则引擎用加权评分（-100~100）给出确定性结论；DeepSeek AI 做语义理解和深度推理。两者互为补充，即使不配 API Key 也能用。

⚡

缓存优先，静默更新

所有数据都先展示缓存 → 后台刷新。用户感知不到等待，页面瞬间可交互。后端 TTL 机制避免频繁调用外部 API。

项目最初为个人投资分析工具开发，核心诉求是低延迟、高可用、易扩展。选择 Flask + Vanilla JS 而非 React/Next.js，原因是桌面应用的维护者通常是个人开发者，技术栈越薄越好——不引入前端框架意味着零依赖升级风险。

03后端设计

Blueprint 模块化

Flask 的 Blueprint 机制让每个功能域独立为一个模块，拥有自己的 URL 前缀和请求处理。所有蓝图通过 shared.py 访问全局状态，而非直接导入变量（避免模块加载时捕获旧引用）。

Blueprint文件核心端点职责

核心层 (core/) — 纯业务逻辑

core/ 包不依赖 Flask，不处理 HTTP 请求。它是纯 Python 业务逻辑，可以被任何框架调用。

config.py

dotenv 加载配置 → 常量（API keys、周期预设、超时、TTL）。设置页修改后热更新内存值。

api_client.py

csqaq.com 封装：K线优先 → chart 回退。内置 429 指数退避重试 (1s/2s/4s)，请求间隔 1s。

indicators.py

pandas 计算 MA5/10/20/60、RSI(14)、MACD、布林带、KDJ、动量、波动率。输入 OHLCV DataFrame。

recommendation.py

7 维度加权评分：趋势(25)、动量(20)、超买超卖(15)、成交量(10)、波动率(10)、支撑阻力(10)、市场情绪(10)。输出 -100~100 分 + buy/hold/sell。

ai_analysis.py

DeepSeek Chat Completions：全量指标数据发送，max_tokens=24576。system prompt 从 PromptManager 模板读取，content 必须是纯字符串。

prompts.py

PromptManager 单例：4 类模板（analysis_system / analysis_instruction / chat_system / portfolio_advice），每类多模板，下拉切换激活，JSON 持久化。

shared.py — 共享状态中心

为什么不用 React/Vue？ 桌面应用的维护者是个人开发者。Vanilla JS 零依赖，不会有框架升级导致的 breaking change。模块按文件拆分，每个 ~200-600 行，通过 <script> 标签加载顺序隐式管理依赖——简单直接，不需要打包器。CSS 同理，原生 Custom Properties 实现主题切换，13 个 CSS 文件各管各的，不会相互干扰。

06持久化策略

数据存储格式读写时机
ID 映射 data/饰品id_*.json {id: name} 启动时加载到内存，运行时可通过设置页上传替换并热更新
自选列表 watchlist.json [{id, name, price, ...}] 每次 CRUD 操作即时读写
自选走势图 watchlist_sparklines.json {item_id: [prices]} 拉取后持久化，启动恢复，打开页面秒渲染
持仓 portfolio.json [{id, name, buy_price, ...}] 每次 CRUD 操作即时读写
库存缓存 inventory.json {steam_id: {data, sparklines, ts}} 拉取后持久化，TTL=300s，启动恢复
库存绑定 inventory_binding.json {steam_id, raw} 绑定/修改时写入
分析历史 history/{aid}.json + .png JSON 记录 + base64 图表 分析完成时写入，启动时恢复（最多 50 条）
提示词模板 prompts.json 4 类，每类多模板 启动加载到 PromptManager 单例，CRUD 即时持久化
知识库 knowledge_base.json [{title, content, tags, date}] CRUD 即时持久化
UI 设置 settings.json {theme, accent, font_size, ...} 修改时写入
API 密钥 .env KEY=VALUE 设置页修改时写入 + 更新 config 内存值
设计原则：所有持久化用扁平 JSON 文件，不引入数据库。原因：本项目是桌面单用户应用，数据量小（几十条自选、几百件库存），JSON 文件的读写足够快，且用户可以直接用文本编辑器查看/修改/备份。

07关键设计决策

7.1 为什么不用数据库？

本项目的所有数据都是单用户、小数据量：几十条自选、几百件库存、几十条分析历史。JSON 文件读写的延迟 < 5ms，完全不需要 SQLite 的查询能力。JSON 对人类可读、可备份、可版本控制，维护成本为零。

7.2 为什么「缓存优先 + 后台刷新」？

外部 API（Steam、csqaq）的延迟不可控（500ms ~ 5s）。如果等数据回来再渲染，用户看到的是白屏。先展示缓存数据（从本地 JSON 文件读取，通常 < 10ms），用户瞬间就能交互；然后后台拉新数据，静默替换 DOM。这个模式贯穿整个前端——库存、自选、走势图都遵循这个原则。

7.3 为什么 5 分钟 TTL？

CS2 饰品价格波动不快（日波动通常 1-5%），Steam 库存也不会频繁变化。5 分钟是「足够实时」和「保护 API 额度」之间的平衡点。TTL 适用缓存，用户手动点按钮 → force=true 绕过。

7.4 为什么提示词做成模板系统？

不同用户有不同的分析偏好：有人关注短线，有人关注收藏价值。把 AI 提示词做成可编辑模板，让用户定制角色设定、分析维度、输出格式，而不需要改代码。每类支持多模板切换，即时生效。builtin 模板不可删保证始终有可用模板。

7.5 为什么 SSE 而非 WebSocket？

AI 追问只需要单向推送（服务端 → 客户端逐 token 发送）。SSE 比 WebSocket 更轻量：不需要握手升级、不需要心跳保活、浏览器原生支持 EventSource。对于这个场景，SSE 是更合适的选择。

7.6 为什么规则引擎 + AI 双轨？

规则引擎是确定性的——同样的指标输入永远得到同样的评分。AI 是非确定性的——有语义理解能力但可能不稳定。两者互补：规则引擎给出量化的买卖信号（-100~100 分），AI 给出定性分析和策略讨论。用户可以不配 AI Key，规则引擎独立工作。

7.7 前端文件为什么按功能拆分 13 个？

一个 ~4000 行的 JS 文件难以维护。拆成 13 个文件，每个 ~200-600 行，功能边界清晰。CSS 同理。加载顺序就是隐式依赖管理（core.js 最先，其他文件依赖 core.js 定义的变量）。没有打包器意味着没有 source map 问题，浏览器 DevTools 可以直接调试源文件。
